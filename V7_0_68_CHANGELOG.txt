V7.0.68 — SAFE jobs lifecycle fix

Scope
- Corrective release focused only on ProcessingJob lifecycle stability.
- No product feature expansion.
- No intentional change to backtests/scenarios/universes behavior.

What was fixed
- Fixed active job control views used by /jobs URLs.
- Cancel now immediately marks a PENDING job as CANCELLED and sets finished_at.
- Cancel on RUNNING jobs now remains cooperative only (cancel_requested=True, status stays RUNNING until task exits).
- Kill now sets both kill_requested=True and cancel_requested=True.
- Kill now immediately marks a PENDING job as KILLED and sets finished_at.
- Kill now performs SIGTERM revoke with SIGKILL fallback when a Celery task_id exists.

Root cause
- The project had two different implementations of job_cancel/job_kill.
- URLs were routed to core/job_views.py, but the more complete behavior existed in core/views.py.
- As a result, the UI actions Cancel/Kill did not apply the expected state transitions.

Validation
- core.tests.test_jobs_robustness passes after the fix.

Operational note
- The manual cleanup command remains available:
  python manage.py cleanup_processing_jobs --older-than-minutes 120 --include-pending --status FAILED
