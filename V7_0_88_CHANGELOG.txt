V7.0.88 — Sprint P1.2 (exports hardening)

Scope
- Fix export CSV Arrow helper to be idempotent and non-destructive.
- Replace remaining direct ws.append(...) calls in views/exports/tasks with append_excel_row(...).
- Lock architecture audit in strict mode by default now that baseline is clean.
- Add baseline test asserting zero architecture violations.

Why
- The quality gate caught a real export regression before prod: the Arrow CSV helper rebuilt already-safe tables.
- Excel exports still had unsafe direct append calls that could reintroduce list/dict/set serialization issues.

Changes
1) Arrow CSV helper
- core/views.py::_arrow_table_to_csv_safe now returns the original object unchanged when no nested Arrow columns need normalization.
- Non-Arrow-like objects are also returned unchanged.

2) Excel export hygiene
- core/views.py: replaced remaining ws.append(...) calls with append_excel_row(...).
- core/exports.py: replaced remaining ws.append(...) calls with append_excel_row(...).
- core/tasks.py: hardened GameScenario XLSX export with append_excel_row(...).

3) Quality gate
- tools/run_quality_gate.py now fails on architecture violations by default.
- New opt-out flag: --allow-architecture-violations

4) Tests
- Added architecture baseline assertion: total violation_count == 0.

Expected validation
- python tools/run_quality_gate.py
- Expected: full success, no architecture violations, export regression tests green.
