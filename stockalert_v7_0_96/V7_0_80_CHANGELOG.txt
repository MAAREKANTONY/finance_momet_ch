STOCKALERT V7.0.80 — Hardening jobs async — Itération 2 (checkpoints réactifs)

Objectif
- rendre les jobs longs plus observables et plus réactifs au Cancel/Kill
- sans changer l’architecture, sans ajouter de worker, sans refacto lourde

Contenu
1) Helper central de cadence des checkpoints
- ajout de core.job_tracking.JobCheckpointPulse
- permet d’émettre job_checkpoint(...) de façon throttlée :
  - tous les X items
  - ou toutes les X secondes
- évite de saturer la DB tout en gardant un heartbeat exploitable

2) Checkpoints ajoutés sur les traitements lourds
- _fetch_daily_bars_for_symbols
  - checkpoint par symbole
  - progression plus lisible dans /jobs
- _compute_metrics_for_scenario
  - checkpoint par symbole
  - checkpoint throttlé dans les boucles de dates
  - checkpoints explicites avant/après enrichissement Global Momentum
- compute_metrics_task / compute_metrics_all_job_task
  - propagation du job courant pour garder un heartbeat pendant les compute all

3) Checkpoints ajoutés sur les exports async
- export_alerts_csv_task
- export_all_scenarios_zip_task
- export_data_xlsx_task
- export_backtest_debug_csv_task
- export_game_scenario_xlsx_task

4) GameScenario runner aligné sur le helper central
- suppression de la logique locale de heartbeat isolée
- run_game_scenario_now utilise désormais JobCheckpointPulse
- checkpoints par phase : start, scenario_synced, bars_ready, depth_checked, metrics_ready, kpi_computed, done
- checkpoints throttlés pendant la construction de today_results

Tests
- ajout de tests ciblés sur JobCheckpointPulse

Impact attendu
- meilleure visibilité du point réel d’avancement
- Cancel/Kill plus réactifs pendant les longues boucles
- moins de faux jobs RUNNING silencieux sans checkpoint utile

Pas encore couvert
- watchdog/audit automatique
- recovery automatique des zombies
- kill OS-level renforcé
- UI “suspect jobs” avancée
