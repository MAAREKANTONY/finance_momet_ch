StockAlert V7.0.95 — Sprint P0.3a (JobCheckpointPulse fix)

Scope
- P0 fix only.
- No refactor.
- No migration.

Changes
- Fixed JobCheckpointPulse so the time-based heartbeat no longer fires immediately on the first hit when every_seconds is set.
- The pulse now starts its elapsed-time window on first observed iteration, then emits:
  - immediately when force=True
  - on the counter threshold (every_n)
  - on the elapsed-time threshold since first hit / last emitted pulse

Root cause
- _last_ts was initialized to 0.0.
- This made the condition `(now_ts - _last_ts) >= every_seconds` true on the very first hit for large monotonic timestamps.
- Result: premature heartbeat emission and failing robustness test.

Files changed
- core/job_tracking.py

- `python tools/run_quality_gate.py` ✅

Impact
- Deterministic pulse behaviour restored.
- Heartbeat/checkpoint writes are no longer emitted too early.
- No functional change outside cooperative job checkpoint throttling.
