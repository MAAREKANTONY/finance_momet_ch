V7.0.97 — Sprint P0.4 Jobs hardening

Objectif
- durcir la stabilité des jobs Game avant d’attaquer les KPI backtest
- éviter les faux multi-running et réduire le temps de présence des zombies RUNNING après cancel/kill

Revue de code préalable
- le moteur de launch était stable, mais il autorisait plusieurs RUN_GAME actifs pour un même GameScenario
- le recovery traitait bien les zombies "classiques", mais trop lentement pour les RUNNING avec cancel/kill demandé
- la tâche planifiée de recovery passait seulement toutes les heures, ce qui laissait longtemps des RUNNING visuellement bloqués
- cancel/kill sur RUNNING posaient bien les flags, mais sans trace explicite côté message opérationnel

Corrections appliquées (safe, minimales, additives)
1) Garde-fou anti doublon sur RUN_GAME
- ajout de core.job_launch.find_active_processing_job(...)
- blocage du lancement d’un nouveau RUN_GAME si un job PENDING/RUNNING existe déjà pour le même GameScenario
- appliqué sur:
  - game_scenario_launch
  - trigger_page (action game_launch / game_force_recompute)

2) Recovery plus rapide des jobs stoppés
- ajout d’un seuil dédié requested_stop_minutes dans core.job_recovery
- un RUNNING avec cancel_requested=True ou kill_requested=True peut maintenant être récupéré beaucoup plus tôt qu’un zombie classique
- cleanup_stale_processing_jobs_task transmet désormais ce seuil
- recover_jobs expose aussi l’option --requested-stop-minutes

3) Watchdog plus fréquent
- cleanup-stale-processing-jobs passe d’une fréquence horaire à une fréquence toutes les minutes dans Celery Beat
- objectif: réduire fortement la durée de visibilité des faux RUNNING après stop demandé

4) Visibilité opérationnelle améliorée
- cancel sur RUNNING ajoute un message explicite: "Cancel requested by user."
- kill sur RUNNING ajoute un message explicite: "Kill requested by user."
- pas de faux passage terminal immédiat: la sémantique reste coopérative et sûre

Tests ajoutés
- lookup du job actif par GameScenario
- blocage du relaunch Game si un job actif existe déjà
- recovery rapide d’un RUNNING cancel_requested
- recovery rapide d’un RUNNING kill_requested

Validation
- python manage.py test core.tests.test_job_launch core.tests.test_jobs_robustness -v 2 ✅
- python tools/run_quality_gate.py ✅

Impact produit
- tu ne pourras plus empiler 4 RUN_GAME simultanément sur le même GameScenario
- après cancel/kill, un job qui ne coopère plus correctement devrait sortir bien plus vite de l’état RUNNING
- l’UI jobs devrait donc devenir beaucoup plus lisible et déterministe

Migration
- aucune migration DB

Recommandation de test manuel
1) lancer un GameScenario
2) relancer immédiatement le même GameScenario
   => le second lancement doit être refusé
3) lancer un Game long puis cancel
   => soit arrêt coopératif rapide, soit recovery watchdog en quelques minutes max, sans rester bloqué longtemps en RUNNING
