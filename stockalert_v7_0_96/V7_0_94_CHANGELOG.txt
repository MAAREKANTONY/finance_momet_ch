V7.0.94 — Sprint P0.2 (Recovery unifiée)

Objectif
- Faire de recover_jobs la source de vérité unique pour la récupération des ProcessingJob stale/zombies.

Modifications
- cleanup_stale_processing_jobs_task délègue désormais à core.job_recovery.recover_jobs
- alignement automatique des statuts FAILED / CANCELLED / KILLED entre recovery manuelle et cleanup planifié
- conservation de la synchronisation des objets métier via sync_related_state_for_terminal_job
- ajout de tests de robustesse pour vérifier :
  - pending + cancel_requested => CANCELLED
  - pending + kill_requested => KILLED
  - stale running backtest => FAILED + backtest synchronisé
- quality gate enrichi avec l’exécution de core.tests.test_jobs_robustness

Impact attendu
- même job stale => même décision, qu’on lance recover_jobs manuellement ou via la tâche planifiée cleanup_stale_processing_jobs_task
- réduction des incohérences visibles dans /jobs pour les jobs zombies

Validation recommandée
1. python tools/run_quality_gate.py
2. créer un job stale RUNNING puis lancer cleanup_jobs
3. créer un job PENDING avec cancel_requested ou kill_requested et vérifier le statut terminal attendu
