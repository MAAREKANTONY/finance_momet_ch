StockAlert V7.0.96 — Sprint P0.3 (sync métier GameScenario)

Objectif
- Étendre la synchronisation métier des jobs terminaux au-delà du Backtest.
- Corriger de façon additive et sûre le cas RUN_GAME pour éviter les GameScenario bloqués en "running" après échec / cancel / kill / recovery.

Revue de code avant correction
1) Source de vérité déjà correcte côté recovery:
   - recover_jobs -> apply_recovery -> sync_related_state_for_terminal_job
   - le point faible n'était pas le moteur de recovery mais le périmètre trop limité du sync métier.
2) RUN_GAME n'avait pas d'objet propriétaire traçable dans ProcessingJob:
   - ProcessingJob ne stockait que backtest/scenario
   - impossible de resynchroniser proprement un GameScenario après crash/recovery de façon déterministe.
3) run_game_scenario_job_task ne rappelait pas sync_related_state_for_terminal_job sur CANCELLED / KILLED / FAILED.
   - impact: état métier potentiellement "running" alors que le job est terminal.
4) Le correctif le plus sûr est additif:
   - ajouter un FK nullable ProcessingJob.game_scenario
   - propager ce lien aux launches RUN_GAME
   - étendre sync_related_state_for_terminal_job sans refacto large.

Corrections livrées
- Ajout du champ nullable ProcessingJob.game_scenario + index.
- launch_processing_job accepte désormais game_scenario.
- Les launches RUN_GAME depuis les vues renseignent maintenant game_scenario.
- Le fallback de création de job dans run_game_scenario_job_task renseigne aussi game_scenario_id.
- sync_related_state_for_terminal_job couvre maintenant:
  - Backtest: CANCELLED/KILLED/FAILED -> Backtest.FAILED + error_message
  - GameScenario: CANCELLED/KILLED/FAILED -> last_run_status + last_run_message
- run_game_scenario_job_task appelle désormais sync_related_state_for_terminal_job sur:
  - GameJobCancelled
  - GameJobKilled
  - Exception
- job_recovery inclut game_scenario_id dans les sélections utilisées pour la resynchronisation terminale.

Tests ajoutés
- sync terminal job -> GameScenario passe à failed + message.
- cleanup/recovery d'un RUN_GAME stale resynchronise le GameScenario.

Validation

- python tools/run_quality_gate.py  OK

Impact / non-régression

- Aucun changement de logique métier Backtest.
- Aucun changement d'export.
- Les anciens jobs sans game_scenario restent compatibles; seul le nouveau flux RUN_GAME devient entièrement resynchronisable.
